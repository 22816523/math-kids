## 2. 背景与目标

### 2.1 项目背景

客户管理是 ERP 系统销售管理模块中的现有功能模块，当前 1.0 版本已经具备单位、联系人、审核、合并、复制、学段年级维护等基础能力，能够支持销售管理中的客户资料录入、审核、处理和维护

从现有代码和页面逻辑来看，1.0 已经形成较完整的客户管理基础能力，主要页面包括单位-联系人管理、单位审核、单位合并、联系人审核、联系人合并、联系人复制等，用户已经建立了现有使用习惯

但当前客户管理仍偏向“单位记录 + 联系人记录”的操作型管理，尚未完全升级为客户主数据治理模式，主要问题包括

- 主客户唯一档案能力不足，不同来源客户数据缺少统一主档治理
- 联系人仍偏业务员名下数据，未完全体现公司级联系人资产能力
- 查重更多依赖后置审核和合并，缺少前置强校验
- 审核与合并偏向操作流程，缺少完整主数据规则支撑
- 多系统同步尚未形成统一映射和日志能力

因此，客户管理 V1.1 的目标不是重做模块，而是在 1.0 基础上，升级为 ERP 销售管理中的客户主数据治理能力

### 2.2 产品目标

| 目标类型 | 具体目标 | 衡量指标 |
| -------- | -------- | -------- |
| 业务目标 | 建立统一主客户档案和统一联系人档案 | 主客户和联系人重复创建率下降 |
| 用户目标 | 业务员可更方便地共享和使用联系人 | 联系人重复录入减少，开通效率提升 |
| 技术目标 | 让客户管理模块成为客户和联系人主数据源 | 主数据同步链路可追踪，可治理 |

### 2.3 项目范围

**本期包含：**
- 主客户档案升级
- 联系人档案升级
- 联系人关系管理
- 主客户查重前置
- 联系人唯一校验前置
- 审核能力增强
- 合并能力增强
- 多系统同步规则设计
- 手机号权限与脱敏控制
- 基础资料维护延续

**本期不包含：**
- 独立客户管理系统建设 - 原因：本项目为 ERP 销售管理模块内迭代升级
- 历史数据迁移项目 - 原因：已确认本次不涉及迁移数据
- 整个 ERP 销售管理模块重构 - 原因：本次仅聚焦客户管理能力升级
- 完整 BI 分析平台建设 - 原因：本期以基础统计和主数据治理为主

## 3. 用户分析

### 3.1 目标用户

| 用户角色 | 特征描述 | 核心诉求 | 使用场景 |
| -------- | -------- | -------- | -------- |
| 业务员 | 日常维护客户和联系人，关注跟进效率 | 快速找到客户、避免重复录入、可共享使用联系人 | 新增客户、维护联系人、查询联系人 |
| 主管 | 关注团队客户资源和联系人分配 | 查看团队资源、管理关系、监督数据质量 | 分配联系人、查看团队客户 |
| 管理员 | 负责数据审核、合并、治理 | 控制数据质量，处理重复数据和异常数据 | 审核、合并、回收、公海管理 |
| 数据维护人员 | 负责同步、治理和异常处理 | 保证主数据一致性和可追踪性 | 同步维护、映射维护、日志跟踪 |

### 3.2 用户画像

**典型用户：销售业务员**
- 基本信息：ERP 销售管理模块日常用户
- 行为特征：频繁查询客户、维护联系人、依赖现有页面完成工作
- 痛点：联系人重复录入、已有联系人无法复用、客户数据来源不统一
- 期望：能基于现有页面快速找到客户和联系人，并便捷建立可用关系

### 3.3 用户旅程

客户录入 → 联系人录入 → 审核处理 → 合并治理 → 业务员使用 → 多系统同步
   ↓         ↓          ↓          ↓         ↓           ↓
 数据分散   重复录入    等待处理     手工治理    使用受限     不可追踪

## 5. 功能详情

### F001 主客户管理

**线框图**

```text
┌──────────────────────────────────────────────────────────────┐
│ 单位-联系人管理                                              │
├──────────────────────────────────────────────────────────────┤
│ [单位状态▼] [单位名称] [省份▼] [城市▼] [区县▼] [单位类型▼]  │
│ [学段▼] [数据来源▼] [类型▼] [联系人]      [查询] [重置]      │
├──────────────────────────────────────────────────────────────┤
│ [新增主客户] [编辑主客户] [启用] [停用] [主客户合并]         │
├──────────────────────────────────────────────────────────────┤
│ 主客户列表                                                   │
│ 编码 | 名称 | 地区 | 单位类型 | 学段 | 数据来源 | 状态      │
├──────────────────────────────────────────────────────────────┤
│ 联系人列表                                                   │
│ 编码 | 联系人 | 手机号 | 业务经理 | 状态                     │
└──────────────────────────────────────────────────────────────┘
```

**功能说明**

在现有单位-联系人管理页基础上，将单位升级为主客户主档，作为 ERP 销售管理中的统一客户单位入口

**数据来源**

- 主客户列表来自客户管理模块主数据表
- 客户名称搜索可复用现有外部搜索接口
- 地区、单位类型、学段等来自现有基础资料接口

**业务规则**

1. 主客户包含全部客户单位，不仅限学校
2. 主客户唯一 ID 以 ERP 客户编码为准
3. 新增主客户前必须执行客户名称 + 地区查重
4. 命中疑似重复客户时，不允许无提示直接提交
5. 学校类和非学校类客户可根据单位类型展示不同字段
6. 主客户审核、合并、修改需要记录日志
7. 主客户是客户主数据源，对外系统以主客户主档为准

**界面要素**

| 元素 | 类型 | 说明 |
| ---- | ---- | ---- |
| 单位状态 | Select | 用于筛选主客户状态，支持未开通、已开通、退回、停用 |
| 单位名称 | Input | 输入主客户名称关键字进行查询 |
| 省份/城市/区县 | Select | 按地区筛选主客户，区县联动上级地区 |
| 单位类型 | Select | 按客户单位类型筛选 |
| 学段 | Select | 学校类主客户筛选条件 |
| 数据来源 | Select | 按主客户来源筛选 |
| 新增主客户 | Button | 打开新增/编辑单位弹窗，进入主客户建档流程 |
| 主客户列表 | Table | 展示主客户基础信息、状态、来源、类型等 |
| 主客户编码 | Link | 点击打开主客户详情弹窗 |

**验收标准**

1. 用户可以在现有单位-联系人管理页看到主客户列表和联系人列表联动结构
2. 新增主客户时系统必须执行查重逻辑
3. 主客户编码以 ERP 客户编码展示
4. 学校类和非学校类客户字段展示规则正确
5. 主客户查询、启停、编辑、合并功能在页面上可正常使用

**异常处理**

| 异常场景 | 处理方式 | 提示文案 |
| -------- | -------- | -------- |
| 查重命中疑似重复客户 | 弹窗提示 | 检测到疑似重复客户，请确认是否关联已有客户 |
| 主客户保存失败 | Toast提示 | 主客户保存失败，请稍后重试 |
| 主客户不存在 | 页面提示 | 当前主客户不存在或已被处理 |
| 权限不足 | Toast提示 | 您暂无该主客户操作权限 |

#### F001-01 主客户查重

**线框图**

```text
┌──────────────────────────────┐
│ 主客户查重结果               │
├──────────────────────────────┤
│ 输入信息：客户名称、地区     │
├──────────────────────────────┤
│ 疑似重复客户列表             │
│ 1. ERP编码 + 名称 + 地区     │
│ 2. ERP编码 + 名称 + 地区     │
├──────────────────────────────┤
│ [继续新建] [关联已有主客户]  │
└──────────────────────────────┘
```

**功能说明**

在新增主客户前对客户名称和地区进行查重，减少重复主客户产生

**业务规则**

1. 查重条件为客户名称 + 地区
2. 地区精度待确认，可能到区县或街道
3. 查重结果展示疑似重复客户
4. 用户可选择继续新建或关联已有主客户
5. 是否允许继续新建需按最终业务规则控制

**界面要素**

| 元素 | 类型 | 说明 |
| ---- | ---- | ---- |
| 疑似重复客户列表 | List/Table | 展示命中的主客户基础信息 |
| 继续新建 | Button | 在规则允许时继续新增主客户 |
| 关联已有主客户 | Button | 直接复用已有主客户 |

**验收标准**

1. 用户输入客户名称和地区后可触发查重
2. 命中重复时系统展示疑似重复客户
3. 用户可以根据规则继续新建或关联已有主客户

**异常处理**

| 异常场景 | 处理方式 | 提示文案 |
| -------- | -------- | -------- |
| 查重服务异常 | Toast提示 | 主客户查重失败，请稍后重试 |
| 无重复结果 | 页面提示 | 未发现重复客户，可继续新建 |

│ 职务        [选择框]                         │
│ 部门        [选择框]                         │

├──────────────────────────────────────────────┤

└──────────────────────────────────────────────┘

**功能说明**

在现有联系人新增、编辑能力基础上，建立统一联系人主档，并在新增时执行唯一性校验

**数据来源**

- 联系人详情来自客户管理模块联系人主档
- 所属主客户信息来自当前选中主客户
- 职务、部门来自基础配置项

**业务规则**

1. 联系人必须填写手机号
2. 联系人唯一规则为主客户 + 姓名 + 手机号
3. 联系人已存在时不重复创建实体，应建立业务员与联系人关系
4. 联系人主档信息和业务员关系信息应区分管理
5. 多个业务员可以同时使用同一联系人
6. 联系人主档修改权限需要按角色和关系控制

**界面要素**

| 元素 | 类型 | 说明 |
| ---- | ---- | ---- |
| 联系人 | Input | 输入联系人姓名，必填 |
| 手机号 | Input | 输入联系人手机号，必填并做格式校验 |
| 业务经理 | SelectPage | 新增联系人时可指定业务经理 |
| 所属主客户 | Input | 展示当前所属主客户，不允许手工随意变更 |
| 职务/部门 | Select | 从基础资料中选择 |
| 状态 | Select | 联系人状态，支持待审核、已启用、退回、停用 |
| 保存 | Button | 保存联系人主档或进入关系建立流程 |

**验收标准**

1. 联系人手机号为空时不可提交
2. 联系人新增时系统执行唯一校验
3. 若联系人不存在，可成功创建联系人主档
4. 若联系人已存在，系统不重复创建联系人实体
5. 联系人详情可正常查看现有基础字段和审核信息

**异常处理**

| 异常场景 | 处理方式 | 提示文案 |
| -------- | -------- | -------- |
| 手机号为空 | 输入框标红 | 请输入手机号 |
| 手机号格式错误 | 输入框标红 | 请输入有效的手机号或座机号 |
| 联系人重复 | 弹窗提示 | 该联系人已存在，请选择开通给当前业务员 |
| 联系人保存失败 | Toast提示 | 联系人保存失败，请稍后重试 |
