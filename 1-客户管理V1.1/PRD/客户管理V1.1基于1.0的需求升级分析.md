# 客户管理 V1.1 基于 1.0 的需求升级分析

## 1 项目背景与升级目标

当前客户管理 1.0 版本已经具备单位、联系人、审核、合并、学段年级维护等基础能力，能够支撑基础客户资料管理和内部流程处理

但从 1.1 版本业务需求来看，需求方希望解决的已经不只是“管理客户资料”，而是要建立公司统一的客户主数据体系，实现不同业务系统间客户与联系人信息的统一、识别、治理和复用

因此，客户管理 V1.1 的定位不应理解为一次普通功能追加，而应理解为

**在现有 1.0 单位与联系人管理能力基础上，升级为统一客户主数据管理平台**

本次升级的核心目标是

- 一个学校，一个档案
- 一个老师，一个档案
- 打通多个业务系统中的客户和联系人信息
- 建立统一的数据标准、查重机制、合并机制、权限机制和同步机制
- 使客户数据从“分散记录”升级为“公司级资产”

---

## 2 1.0 现状能力盘点

结合现有 1.0 原型和表设计，当前系统已经具备以下基础能力

### 2.1 已有页面与功能能力

1. 单位-联系人管理
2. 单位审核
3. 单位合并
4. 联系人审核
5. 联系人合并
6. 联系人复制
7. 学段年级维护

从现状上看，1.0 已经覆盖了客户管理的基本作业场景，即

- 单位信息维护
- 联系人信息维护
- 数据审核
- 数据合并
- 学段、年级基础字典维护

### 2.2 已有数据模型基础

从现有表设计可看出，系统已经具备以下数据基础

- 单位主表
- 单位扩展信息表
- 联系人表
- 学段表
- 年级表
- 学段年级关系表
- 单位学段年级学生数关系表
- 单位类型表及单位类型与业务群关系表

这说明 1.0 已经有较完整的客户资料承载结构，尤其在“单位基础信息 + 联系人基础信息 + 学段年级信息”这三块已有较强基础

### 2.3 1.0 当前能力边界

虽然 1.0 已经有客户管理基础，但整体上仍偏向单系统内部管理工具，主要特征是

- 单位和联系人仍然偏业务操作对象，而非统一主数据对象
- 合并、审核、维护是已有能力，但更多是后台处理逻辑
- 缺少跨业务系统统一标识
- 缺少外部系统映射关系
- 缺少公司级联系人资产管理视角
- 缺少完整的归属权、公海、转移、分配等机制
- 缺少系统级同步治理能力
- 缺少主数据平台层面的安全与分析能力

因此，1.0 可以作为 V1.1 的基础，但不能直接满足 V1.1 的目标

---

## 3 1.1 核心升级方向

### 3.1 从“单位管理”升级为“主客户管理”

1.0 中的单位，本质上是业务中的客户单位记录

到了 1.1，单位需要升级为“主客户档案”，并承担跨系统统一主档角色

这意味着主客户不再只是某个页面中的单位记录，而是所有业务系统识别学校客户的统一中心

### 3.2 从“联系人管理”升级为“公司级联系人资产管理”

1.0 中联系人更偏向业务员使用对象

到了 1.1，联系人被定义为公司级资产，业务员拥有的是跟进权、归属权或使用权，而不是联系人数据本身所有权

这意味着联系人档案必须从“个人名下资源”升级为“公司级统一资源池中的对象”

### 3.3 从“单系统数据维护”升级为“多系统数据打通”

1.1 明确提出涉及多个系统

- 智慧九科星
- 中课云
- 研究院稿费报销
- 推广会相关数据来源

这说明 V1.1 的重点之一，是建立统一客户和联系人主档，并将多个业务系统中的数据映射、归档和同步到主数据库中

### 3.4 从“功能流程”升级为“数据治理体系”

1.1 不只是要补几个页面，而是要求形成完整治理闭环，包括

- 标准化字段
- 全库查重
- 合并规则
- 审核规则
- 权限规则
- 同步规则
- 脱敏规则
- 报表分析能力

这已经属于主数据治理范畴

## 4 复用项 / 改造项 / 新增项分析

## 4.1 复用项

以下内容建议直接复用 1.0 既有能力，并在必要时做轻量优化

### 一 单位基础信息管理能力
- 单位列表查询
- 单位详情维护
- 单位状态管理
- 单位审核流程

### 二 联系人基础管理能力
- 联系人列表查询
- 联系人详情维护
- 联系人审核流程

### 三 数据治理基础能力
- 单位合并
- 联系人合并
- 学段年级维护
- 基础分页、查询、筛选能力

这些能力在 V1.1 中仍然需要，没必要推翻

## 4.2 改造项

以下内容不是新增独立功能，而是在 1.0 基础上做模型升级和规则升级

### 一 单位模型改造为主客户模型
改造重点包括

- 单位编号升级为主客户唯一标识
- 单位记录不再只是单来源记录，而是统一主档
- 增加多系统映射关系
- 增加跨系统同步能力
- 增加主客户标准字段体系

### 二 联系人模型改造为统一联系人模型
改造重点包括

- 联系人以“所属学校 + 姓名 + 手机号”进行唯一识别
- 联系人档案进入公司级数据库
- 业务员拥有的是归属关系，不是数据所有权
- 增加联系人来源、使用部门、归属状态等管理信息

### 三 原查重逻辑升级为强制全库查重
改造重点包括

- 主客户创建时按“客户名称 + 地区”模糊匹配
- 联系人创建时按“所属学校 + 姓名 + 手机号”全库强制检索
- 不再允许简单重复入库
- 对查重结果给出明确处理动作

### 四 原合并逻辑升级为主数据治理逻辑
改造重点包括

- 合并不仅影响当前页面数据
- 合并结果需作用到主档层
- 合并后需同步回传各关联系统
- 合并过程需保留日志和映射关系

### 五 权限逻辑升级
改造重点包括

- 从“页面操作权限”升级为“数据查看权 + 归属权 + 分配权 + 管理权”
- 业务员默认仅可查看和操作自己负责的联系人
- 管理层可看团队或全公司数据
- 公司级搜索与数据查看需配合脱敏规则

## 4.3 新增项

以下内容属于 1.0 基础上明显缺失，V1.1 必须新增的能力

### 一 主客户统一档案能力
- 全局唯一主客户 ID
- 多系统客户映射关系
- 主客户标准字段中心
- 主客户合并归档机制

### 二 公司级联系人资产能力
- 联系人统一档案
- 联系人归属分配机制
- 联系人转移机制
- 联系人复制机制与边界规则
- 联系人公海池机制

### 三 全局搜索能力
- 按学校、姓名、电话进行全局搜索
- 支持跨归属范围的授权查询
- 敏感信息脱敏显示

### 四 数据同步能力
- 各业务系统更新联系人或主客户信息时，支持接口反向同步
- 主数据库作为标准源头之一
- 明确同步方向、同步字段、同步触发时机

### 五 数据安全与分析能力
- 手机号脱敏
- 导出权限限制或不开放导出
- 客户分布统计
- 联系人增长统计
- 客户与联系人全景可视化

## 5 关键业务规则梳理

## 5.1 主客户唯一规则

- 每个学校必须只对应一个主客户档案
- 主客户档案必须有全局唯一 ID
- 多个业务系统中的学校记录必须能够挂接到同一主客户下
- 创建新主客户时必须先做查重提示
- 如识别为同一学校，应优先关联已有主客户，而不是重复创建

## 5.2 联系人唯一规则

- 联系人唯一识别维度为所属学校 + 姓名 + 手机号
- 任意业务员创建联系人时，必须全库检索
- 如联系人已存在，应执行“开通给业务员”而不是重复入库
- 如联系人不存在，才进入审核并同步进库

## 5.3 联系人归属规则

- 联系人属于公司资产
- 业务员仅拥有跟进权、归属权或使用权
- 联系人应记录来源部门和使用部门
- 管理员应支持批量或单独分配联系人
- 联系人支持转移、复制、公海回收

## 5.4 合并规则

- 主客户合并后，应形成唯一主档
- 联系人合并后，不应影响原有业务员正常使用
- 合并动作必须留痕
- 合并结果应同步影响相关业务系统或映射关系

## 5.5 安全规则

- 手机号等敏感信息查询时脱敏显示
- 中间四位隐藏
- 非授权用户不可查看完整敏感信息
- 原则上不开放导出权限

## 6 对页面与数据模型的影响

## 6.1 对页面的影响

### 一 单位-联系人管理页
这是 V1.1 变化最大的页面，建议从原“单位-联系人管理”升级为“主客户-联系人管理”

主要变化包括

- 单位概念升级为主客户概念
- 增加主客户唯一标识展示
- 增加查重提示和关联已有档案入口
- 增加来源系统信息和映射信息
- 联系人增加归属权、来源部门、使用部门、状态信息
- 支持从全局联系人池中开通到业务员
- 支持公司级搜索入口

### 二 单位审核页
需增强为主客户审核页

- 审核不只是本系统入库
- 审核通过后要进入主数据库
- 审核结果需支持同步回传相关系统

### 三 单位合并页
需增强为主客户合并页

- 合并对象不只是两条单位数据
- 需要考虑外部系统映射合并
- 需要保留主次档关系和合并日志

### 四 联系人审核页
需增强查重前置逻辑和审核后的同步逻辑

### 五 联系人合并页
需支持公司级联系人合并，并兼容多业务员使用场景

### 六 联系人复制页
1.0 已有联系人复制能力，V1.1 中可继续保留，但要明确
- 复制的是使用关系还是联系人实体
- 是否形成新的归属关系
- 是否会引发重复档案

### 七 学段年级维护页
可基本复用，但要确保其数据模型适配主客户标准字段体系

## 6.2 对数据模型的影响

建议重点新增或增强以下模型

### 一 主客户主档模型
建议新增或明确以下字段
- 主客户唯一 ID
- 主客户名称
- 省、市、区县、街道、地址
- 单位类型
- 类型
- 学段
- 学生数
- 学校性质
- 单位状态
- 审核状态
- 来源系统
- 创建时间、更新时间
- 审核人、审核时间

### 二 主客户系统映射模型
建议新增
- 主客户 ID
- 来源系统编码
- 外部系统客户 ID
- 同步状态
- 最近同步时间
- 是否主映射

### 三 联系人主档模型
建议增强
- 联系人唯一 ID
- 主客户 ID
- 姓名
- 手机号
- 科目
- 身份证号
- 职务
- 年级
- 地址
- 收货人
- 来源系统
- 来源部门
- 使用部门
- 状态
- 审核信息

### 四 联系人归属关系模型
建议单独建模，不要把所有权和实体混在一起
- 联系人 ID
- 业务员 ID
- 归属类型
- 使用状态
- 来源部门
- 分配时间
- 转移时间
- 回收状态
- 是否公海

### 五 数据同步日志模型
建议新增
- 同步对象类型
- 主档 ID
- 来源系统
- 目标系统
- 同步方向
- 同步结果
- 失败原因
- 同步时间

### 六 合并日志模型
建议新增
- 合并类型
- 主档 ID
- 被合并档案 ID
- 合并人
- 合并时间
- 合并影响范围
- 回传状态

## 7 后续 PRD 编写建议

为了避免把 V1.1 写成“零散加功能”，建议后续 PRD 按“平台升级”方式来写

推荐写法如下

### 一 先定义版本目标
明确 V1.1 是在 1.0 基础上，建设客户主数据统一平台

### 二 再按能力域拆分需求
建议拆成以下模块

1. 主客户档案管理
2. 联系人档案管理
3. 查重与唯一识别
4. 审核与合并治理
5. 联系人归属与分配
6. 全局搜索与权限控制
7. 跨系统映射与同步
8. 数据安全与报表分析

### 三 每个模块中都区分
- 复用原能力
- 改造原能力
- 新增能力

这样研发、测试、产品、业务都会更容易理解范围

---

## 8 结论

客户管理 V1.1 的本质，不是对 1.0 增加几个页面或字段，而是基于现有单位和联系人管理能力，向“统一客户主数据平台”升级

1.0 提供了较好的基础，包括单位、联系人、审核、合并、学段年级和相关表结构

1.1 则在此基础上，进一步提出了以下核心升级要求

- 主客户唯一档案
- 联系人唯一档案
- 公司级联系人资产
- 跨系统打通
- 强制查重与合并治理
- 归属权与权限控制
- 脱敏与安全控制
- 报表与全景分析

因此，后续无论是写 PRD、画原型还是拆研发任务，都应围绕“基于 1.0 升级为客户主数据平台”这个总逻辑展开，而不是按照普通页面增补方式推进
