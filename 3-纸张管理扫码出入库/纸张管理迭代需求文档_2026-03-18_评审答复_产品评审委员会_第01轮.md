## 逐条答复

### 意见 1：增加件级占用/预留机制，避免提纸单和退纸单双重承诺

**原文**：
> 没有设计“件级占用/预留”机制，提纸单和退纸单存在双重承诺风险。

**分类**：风险提示

**答复**：部分采纳

**理由**：

评审意见成立，当前 PRD 对“多单同时盯同一批可用库存”的风险描述不够充分，确实会导致现场才暴露冲突。  
但本期不建议直接升级为“申请时按件锁定”的重方案，因为这会显著改变现有提纸、退纸申请操作方式，也会提高业务录单复杂度，与“本期不改变主流程”的原则冲突。

本期拟采用折中方案：

1. 提纸单、退纸单仍按现有方式申请，不在申请阶段选定具体件。
2. 在审核/核对通过后，对对应仓库、对应纸张规格的可用件数进行冻结校验，避免数量层面被重复承诺。
3. 在发纸/退纸确认节点，再通过扫码将具体件绑定到当前单据，形成件级占用与流转。
4. 已进入“已发出待收货”状态的件，不允许再次被其他单据使用。

该方案能控制核心风险，同时避免本期把申请入口改得过重。

**行动项**：
- [ ] 在 PRD 中补充“数量冻结 + 发货/退货时绑定具体件”的规则说明
- [ ] 增加可用数量校验、冻结释放、状态占用的测试用例

### 意见 2：明确上层汇总字段与件明细的主从关系

**原文**：
> 件级与上层汇总字段并存，但没有定义谁是最终账。

**分类**：事实性错误

**答复**：完全采纳

**理由**：

该问题是本轮必须修正项。若不明确主从关系，开发会出现双维护，测试也无法界定哪个口径才是最终正确结果。

本期拟明确如下：

1. 单据上层字段中的“件数、总吨数”在件明细确认后，以件明细自动汇总结果为准。
2. 上层字段在前置录单阶段可录入，用于业务初始登记与审核参考。
3. 入库确认、发纸确认、收纸确认等关键落账节点后，不再允许上层件数和总吨数脱离件明细独立维护。
4. “总令数”因当前未拆到件级字段，本期继续保留在上层字段维护，不作为件级自动汇总项。

**行动项**：
- [ ] 在 PRD 中补充“件数、总吨数以件明细汇总为准”的主从规则
- [ ] 在测试用例中增加上下层数据一致性校验

### 意见 3：补打规则不完整，存在重复标签风险

**原文**：
> 二维码补打设计不完整，存在重复标签风险。

**分类**：风险提示

**答复**：完全采纳

**理由**：

评审意见成立。系统中的“一物一码唯一”不等于现场一定唯一，如果补打规则缺失，仓库作业会出现“一件纸多张标签”的风险。

本期拟补充规则如下：

1. 补打仅针对原唯一编号重新打印，不生成新编号。
2. 系统记录补打日志，包括补打人、补打时间、补打单据或来源场景。
3. 本期不限制补打次数，但每次补打都必须留痕。
4. 现场操作规范要求旧标签作废，以最新可识别标签为准；系统层面仍只识别同一唯一编号。

**行动项**：
- [ ] 在 PRD 中增加补打规则和补打日志要求
- [ ] 在测试用例中增加补打后扫码一致性验证

### 意见 4：补充随货同行单到货差异处理

**原文**：
> 随货同行单没有覆盖到货差异处理，MVP 会卡在最常见的现场异常上。

**分类**：风险提示

**答复**：部分采纳

**理由**：

评审指出的是典型一线场景，必须补。  
但本期不建议把采购异常协同扩展成完整的差异审批体系，否则范围会快速蔓延到采购、供应商协同、财务对账等模块。

本期拟补充轻量差异处理：

1. 支持短到、超到、破损拒收、分批到货的异常登记。
2. 单据存在差异时，不允许直接按原计划强行完成件级入库。
3. 可先登记异常，再由业务侧按现有流程处理改单或补单。
4. 本期不新增复杂差异审批流，仅把异常留痕和入库拦截规则写清。

**行动项**：
- [ ] 在 PRD 中增加“到货差异处理”章节
- [ ] 明确差异登记后的单据状态和入库限制

### 意见 5：明确提纸单是按件申请还是按规格申请、发货时实配

**原文**：
> 提纸单“按件申请”还是“按规格申请、发货时实配”没有定清楚。

**分类**：方案建议

**答复**：部分采纳

**理由**：

该意见非常关键，当前 PRD 的写法确实容易让开发产生两种理解。  
本期明确选择：

1. 提纸单仍按现有业务习惯，按纸张规格/数量申请。
2. 不在申请阶段选择具体件。
3. 在调出仓确认发纸时，通过扫码对具体件进行实配绑定。

不采纳“申请时按件选择”的原因是：

1. 与当前业务主流程偏差过大。
2. 会增加申请端录单成本，影响业务接受度。
3. 历史库存与现场标签准备不完整时，申请端按件选取会放大实施难度。

**行动项**：
- [ ] 在 PRD 中删除含糊表述，明确“按规格申请、发货时实配”
- [ ] 在流程图和业务规则中统一口径

### 意见 6：明确件级字段可编辑边界

**原文**：
> 件级字段的可编辑边界没定，追溯可信度不够。

**分类**：风险提示

**答复**：完全采纳

**理由**：

该意见成立。一物一码体系的核心不是“能录进去”，而是“录进去之后可信”。  
如果件级关键字段在后续节点可随意修改，追溯体系会失效。

本期拟明确：

1. 唯一编号一旦生成并正式入账，不允许修改。
2. 吨数、出厂编码、照片在首次正式入账前允许补充或修正。
3. 首次正式入账后，吨数默认不可编辑；若确需调整，需走特殊调整动作并留痕。
4. 出厂编码和照片如需补录，可保留受控修改能力，但必须记录修改人、时间和修改前后值。

**行动项**：
- [ ] 在 PRD 中补充件级字段可编辑矩阵
- [ ] 增加字段修改留痕要求

### 意见 7：补充权限点清单

**原文**：
> 权限“不写死”不等于“可以不定义”。

**分类**：风险提示

**答复**：完全采纳

**理由**：

认同该意见。权限不写死到代码逻辑，不代表 PRD 可以不定义权限颗粒。  
如果没有权限点清单，实施、测试、权限配置都会失去依据。

本期将在 PRD 中至少列出以下权限点：

1. 一物一码生成
2. 二维码批量打印
3. 二维码补打
4. 手工录码
5. 库位主数据维护
6. 历史库存初始化导入
7. 一物一码查询
8. 件级特殊调整

**行动项**：
- [ ] 在 PRD 中补充权限点清单
- [ ] 在测试范围中增加权限校验场景

### 意见 8：增加历史库存初始化后的对账要求

**原文**：
> 历史库存初始化缺少与现有库存汇总账的对账要求。

**分类**：风险提示

**答复**：完全采纳

**理由**：

评审意见成立。初始化不只是“导入成功”，更关键的是导入后件级库存和现有纸张层库存必须能对得上，否则后续所有出入库都会在基础账上出问题。

本期拟补充：

1. 初始化前，对既有汇总库存数据进行基线确认。
2. 初始化导入后，按仓库、纸张规格、件数、总吨数维度进行对账。
3. 对账不一致的数据不得视为初始化完成，需要输出差异清单。
4. 差异清单由业务侧核对后，才能进入正式件级流转。

**行动项**：
- [ ] 在 PRD 中新增“初始化对账要求”
- [ ] 增加初始化结果对账报表或导出要求

### 意见 9：补充手机端扫描页的进度反馈设计

**原文**：
> 手机端操作步骤已经很重，但缺少批量反馈和进度感设计。

**分类**：方案建议

**答复**：完全采纳

**理由**：

该意见直接关系一线可用性。本项目的真实考验不在后台，而在仓管扫几十件、上百件时是否还能顺畅完成任务。

本期将在交互层补充以下能力：

1. 已扫件数 / 应扫件数实时展示。
2. 未完成件数提示。
3. 异常件清单提示。
4. 未填库位件数提示。
5. 提交按钮前置校验反馈。

**行动项**：
- [ ] 在 PRD 中补充手机端扫描页交互要求
- [ ] 在后续原型或交互稿中重点体现进度反馈

### 意见 10：补充一物一码查询后的跳转动作

**原文**：
> 一物一码查询有价值，但没有定义“查到之后能做什么”。

**分类**：方案建议

**答复**：部分采纳

**理由**：

认同该建议。一物一码查询如果只能“看”，价值会被削弱。  
但本期也不建议把查询页做成复杂的操作中台，以免范围继续扩张。

本期拟补充的能力：

1. 支持从查询结果跳转到来源单据。
2. 支持跳转到最近一次业务单据。
3. 显示当前仓库和库位定位信息。

本期暂不强制补充：

1. 从查询结果直接发起复杂业务操作。
2. 独立库位详情页的深度展开。

**行动项**：
- [ ] 在 PRD 中补充查询页跳转能力说明
- [ ] 统一 PC 端和手机端查询结果的跳转口径
