## 2. 背景与目标

### 2.1 项目背景

公司主营图书销售，图书为自研产品，并委托不同印刷厂进行生产印刷。纸张由公司统一采购并提供，纸张会存放于公司多个物理仓，同时各印刷厂现场也存在纸张实物存放场景。围绕纸张实物流转，当前 ERP 已支持以下业务单据流程：

- 随货同行单：用于采购纸张入库到公司仓库。
- 提纸单：用于纸张调拨，覆盖印厂到公司、印厂到印厂、公司仓到公司仓等场景。
- 退纸单：用于印厂将纸张退回公司。

当前系统在随货同行单、提纸单、退纸单中，仅管理到纸张品类/规格层级，例如“65克870*1230卷筒双胶纸岳阳”，记录的是纸张属性、件数、总吨数、总令数、入库仓库等信息，但未对每一卷或每一件纸张单独管理。实际业务中，每一件纸张存在以下差异：

- 单件吨数不一致。
- 供应商可能为单件纸张贴有出厂编码。
- 同一纸种下每件纸张需要单独追踪流转位置与状态。

同时，当前库存仅管理到“仓库”层级，尚未实现“库位”管理。随着纸张积压增加，仓管依赖个人经验找货，一旦相关人员请假或交接不足，其他人员难以快速定位纸张实物，直接影响出入库效率与准确性。

因此，本次迭代需在不推翻现有三类单据业务流程的前提下，补齐“件级管理、一物一码、库位管理、扫码出入库”能力。

### 2.2 当前问题

| 问题类型 | 现状描述 | 业务影响 |
| ---- | ---- | ---- |
| 件级管理缺失 | 单据只管理到纸张层，不管理到单卷/单件 | 无法做到单件追踪，库存精度不足 |
| 无一物一码 | 每件纸张没有系统内唯一身份标识 | 无法扫码流转、无法快速核对、无法追溯 |
| 无库位管理 | 入库仅到仓库，不到具体库位 | 找货依赖人，盘点和交接效率低 |
| 出入库核对粗放 | 按件数/总量核对，缺少逐件校验 | 错发、漏发、错收风险较高 |
| 历史库存未纳入系统件级追踪 | 已在线下人工补码并 Excel 维护 | 数据分散，系统与现场脱节 |

### 2.3 本期目标

| 目标类型 | 具体目标 | 衡量方式 |
| ---- | ---- | ---- |
| 业务目标 | 实现纸张件级管理，一件纸对应一个系统唯一编号 | 三类单据均可按件明细完成流转 |
| 业务目标 | 实现扫码出入库，提升实物流转核对准确性 | 出入库/调拨/退纸环节支持逐件扫码 |
| 业务目标 | 实现仓库内库位管理，支持快速定位实物 | 每件在库纸张均可定位到仓库+库位 |
| 管理目标 | 支持按一物一码查询纸张当前状态与历史轨迹 | 系统支持按唯一编号查询 |
| 系统目标 | 将历史库存补码数据初始化进系统 | 支持历史库存初始化导入 |

### 2.4 本期范围

**本期包含**

- 随货同行单增加件明细管理。
- 提纸单增加件明细管理。
- 退纸单增加件明细管理。
- 新增纸张一物一码编码、二维码标签打印能力。
- 新增纸张扫码入库、扫码出库、扫码收货能力。
- 新增库位管理能力，并在件级库存中记录仓库与库位。
- 新增一物一码查询能力。
- 新增历史库存初始化能力。
- 兼容现有 PC 端与手机端单据流程，在关键实物核对节点增加扫码操作。

**本期不包含**

- 重构现有三类单据的整体审批流程。
- 对供应商出厂编码建立统一校验规则。
- 在业务代码中写死角色权限规则，权限仍由云平台菜单/角色体系配置。
- 将二维码标签打印为复杂模板内容，本期仅要求打印二维码和唯一编号。
- 将库位扩展为多级结构，本期仅支持“仓库 + 库位”两级。

### 2.5 产品原则

1. 不改变现有业务主流程，仅在涉及实物流转的关键节点增加扫码与件级数据。
2. 系统中所有纸张件级流转必须以唯一编号为核心，不允许同一编号对应多件纸。
3. 所有入库完成后的在库纸张，必须具备明确仓库和库位。
4. 所有出入库核对环节以逐件扫码为主，手工输入二维码编号为扫码失败时的补充方式，且输入结果必须与扫码识别结果口径一致。
5. 提纸单、退纸单不允许部分发货、部分收货、部分退货。

## 3. 用户分析

### 3.1 目标用户

| 用户角色 | 角色说明 | 核心诉求 | 典型场景 |
| ---- | ---- | ---- | ---- |
| 收货仓库仓管 | 负责公司仓纸张收货、入库、库位落位 | 快速收货、准确入库、能找到纸 | 随货同行单入库、退纸收货 |
| 调出仓库仓管 | 负责按提纸单发纸 | 快速核对待发纸张，避免错发漏发 | 提纸单发纸 |
| 调入仓库仓管 | 负责按提纸单收纸入库 | 快速核对收货，明确库位 | 提纸单收纸 |
| 印厂退纸人员 | 负责按退纸单退还纸张 | 清晰知道退哪些纸、确保实物一致 | 退纸单退纸 |
| 公司纸张管理员 | 负责纸张业务审核和统筹 | 可追踪、可核对、可追责 | 提纸单核对、退纸申请、追溯查询 |
| 纸张管理人员 | 负责历史库存补码、初始化准备 | 能把线下补码数据导入系统 | 历史库存初始化 |

### 3.2 用户痛点

**收货/仓管侧痛点**

- 只能按件数和总吨数收货，无法确认每件实物到底是哪一件。
- 收纸后无法在系统中记录到具体库位，后续找货困难。
- 交接依赖个人经验，替班人员难以定位库存。

**管理侧痛点**

- 无法按单件追踪纸张从采购入库、调拨、退回的完整流转。
- 发现问题时无法快速判断某件纸当前在哪个仓、哪个库位。
- 历史库存靠 Excel 管，线上线下信息割裂。

**印厂侧痛点**

- 提纸、退纸时缺少清晰的件级核对方式。
- 容易出现“数量对了，但件不对”的问题。

### 3.3 核心使用场景

1. 采购纸张到货后，公司仓管根据随货同行单，在入库确认环节逐件扫码或录码，补充件明细与库位，完成收货入库。
2. 调拨提纸时，调出仓按提纸单逐件扫码发纸，调入仓逐件扫码收纸并录入库位，完成调拨闭环。
3. 印厂退纸时，印厂按退纸单逐件扫码退纸，公司仓管逐件扫码收纸并录入库位。
4. 管理人员根据唯一编号查询某件纸的当前仓库、当前库位、来源单据及流转轨迹。
5. 历史库存由纸张管理人员在线下补码后，按初始化模板导入系统，纳入后续统一件级管理。

### 4.2 核心方案说明

#### 4.2.1 最小管理单元

- 卷筒纸：按“一卷”作为一件管理。
- 平板纸：按“一件”作为一件管理。
- 系统内统一抽象为“纸张件明细”。

#### 4.2.2 一物一码规则

- 编码格式：`年份 + 6位流水号`。
- 示例：`2026000001`。
- 流水号按年份递增。
- 每个自然年从 `000001` 重新开始。
- 唯一编号在系统内全局唯一，同一年内不得重复。

#### 4.2.3 二维码标签规则

- 标签内容仅包含：二维码、唯一编号。
- 标签打印为本期范围内能力。
- 二维码识别结果与手工输入的唯一编号口径一致。
- 标签贴附时机：在入库前完成生成和打印，并贴于对应纸张实物。

#### 4.2.4 库位规则

- 库位模型为“仓库 + 库位”两级。
- 一件纸张在入库完成后必须落到明确库位。
- 调入收货、退纸收货等再次入库场景，均需重新确认库位。

#### 4.2.5 扫码规则

- 仅在涉及实物出入库核对的节点要求扫码。
- 扫码失败时支持手工输入唯一编号。
- 手工输入后，系统校验该编号是否存在、是否符合当前单据业务要求。
- 不允许使用未建码纸张完成正式件级流转。

#### 4.2.6 历史库存处理方案

- 历史库存由纸张管理人员先在线下打印二维码并贴码。
- 通过 Excel 维护一物一码及初始化基础信息。
- 系统提供初始化导入能力，将历史库存导入件级库存体系。
- 历史库存初始化完成后，纳入统一扫码出入库管理。

## 5. 功能详情

### F001 随货同行单件级入库管理

**功能说明**

在不改变现有随货同行单主流程的前提下，为随货同行单增加件明细、一物一码、二维码打印、扫码入库和库位落位能力，实现采购纸张从到货到入库的件级管理。

**适用流程**

收货仓库的仓管新增随货同行单（拍照上传） → 提交申请 → 编辑单据（补充纸张和采购相关信息） → 提交审核 → 审核随货同行单 → 仓管确认入库 → 系统自动生成入库单

#### 5.1 页面/节点改造范围

| 节点 | 端 | 改造内容 |
| ---- | ---- | ---- |
| 新增随货同行单 | 手机/PC | 保持现有逻辑，不强制录入件明细 |
| 编辑单据 | PC | 增加件明细入口、二维码打印入口 |
| 审核随货同行单 | PC | 可查看件明细汇总信息 |
| 仓管确认入库 | 手机优先，PC兼容 | 逐件扫码/录码、维护件明细、录入库位、确认入库 |
| 自动生成入库单 | 系统 | 生成入库单时同步写入件级库存记录 |

#### 5.2 件明细字段

| 字段 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| 唯一编号 | Text | 是 | 一物一码，格式为年份+6位流水 |
| 吨数 | Number | 是 | 单件纸张吨数 |
| 出厂编码 | Text | 否 | 供应商单件纸张编码，不做唯一性限制 |
| 库位 | Select | 入库确认时必填 | 纸张落位的具体库位 |
| 照片 | Image | 否 | 单件纸张照片，可用于辅助识别 |

#### 5.3 业务规则

1. 随货同行单的纸张明细仍保留原有“纸张品类/规格、件数、总吨数、总令数”等上层信息。
2. 在件级维度下，每个纸张明细行需补充对应数量的件明细记录。
3. 件明细数量必须与单据中的件数一致，才允许完成入库确认。
4. 唯一编号必须存在且不可重复使用于其他件明细。
5. 二维码需在入库前完成生成和打印，并贴附到对应实物上。
6. 仓管确认入库时，必须对实际入库的每件纸张进行扫码或手工录码。
7. 每件纸张入库时必须录入库位，未录入库位不得完成入库。
8. 手工输入唯一编号时，系统按与扫码结果一致的口径进行校验。
9. 出厂编码可为空，不校验唯一性，不作为业务拦截条件。
10. 件明细中的照片非必填，不作为入库拦截条件。

#### 5.4 状态流转

| 状态 | 触发条件 | 可执行操作 |
| ---- | ---- | ---- |
| 待建件明细 | 单据通过审核，未完成件级准备 | 生成编号、打印标签、补充件明细 |
| 待入库扫码 | 件明细已准备完成，待仓管确认入库 | 扫码、录码、录入库位 |
| 已入库 | 仓管完成全部件级扫码并确认入库 | 查看入库结果、生成入库单 |

状态流转图：

```text
待建件明细 -> 待入库扫码 -> 已入库
```

#### 5.6 仓管确认入库流程

1. 仓管进入待入库的随货同行单。
2. 系统展示该单据应入库的纸张明细及应入库件数。
3. 仓管按实物逐件扫码，系统识别唯一编号并匹配件明细。
4. 如扫码失败，可手工输入唯一编号完成识别。
5. 仓管为每件纸录入或确认吨数、出厂编码、库位，照片可选上传。
6. 当所有应入库件明细均已完成识别且库位完整后，允许点击“确认入库”。
7. 系统将件级库存写入在库状态，记录当前仓库、当前库位、来源单据为随货同行单。
8. 系统自动生成入库单。

#### 5.7 验收标准

1. 随货同行单审核通过后，可生成该单件级二维码标签并打印。
2. 仓管确认入库时，系统可逐件扫码或手工录入唯一编号。
3. 件明细数量不足、超出、重复时，系统阻止入库。
4. 任一件未填写库位时，系统阻止入库。
5. 入库完成后，可在件级库存中查询到每件纸的仓库、库位、来源单据。
6. 系统自动生成的入库单与件级入库结果一致。

#### 5.8 异常处理

| 异常场景 | 处理方式 | 提示文案 |
| ---- | ---- | ---- |
| 扫码识别失败 | 支持手工录码 | 二维码识别失败，请重新扫描或手工输入唯一编号 |
| 输入编号不存在 | 拦截提交 | 该唯一编号不存在，请确认后重试 |
| 输入编号重复 | 拦截提交 | 该唯一编号已在本单据中使用，请勿重复录入 |
| 件明细数量不足 | 拦截入库 | 当前已录入件数不足，不能确认入库 |
| 件明细数量超出 | 拦截录入 | 当前录入件数已超过单据件数，请检查 |
| 未填写库位 | 拦截入库 | 存在未填写库位的纸张件，请完善后再入库 |
| 同一码已在库 | 拦截提交 | 该唯一编号已存在库存记录，不能重复入库 |

### F002 提纸单件级调拨管理

**功能说明**

在现有提纸单流程中增加件级扫码发纸、扫码收纸、库位落位能力，实现纸张调拨过程的逐件核对与轨迹追踪。

**适用流程**

调入仓库提交申请 → 公司纸张管理员核对 → 调出仓库确认发纸 → 公司管理员初审 → 调入仓库确认收纸 → 系统自动生成调拨单

#### 5.9 页面/节点改造范围

| 节点 | 端 | 改造内容 |
| ---- | ---- | ---- |
| 调入仓库提交申请 | 手机/PC | 保持现有申请逻辑，可查看可调拨件信息或按原方式申请 |
| 公司纸张管理员核对 | PC | 可查看拟调拨件明细 |
| 调出仓库确认发纸 | 手机优先，PC兼容 | 逐件扫码出库 |
| 公司管理员初审 | PC | 可查看发纸件明细结果 |
| 调入仓库确认收纸 | 手机优先，PC兼容 | 逐件扫码收纸并录入库位 |
| 自动生成调拨单 | 系统 | 生成调拨单并更新件级库存 |

#### 5.10 业务规则

1. 提纸单需按件维度确认实际调拨的纸张。
2. 调出仓确认发纸时，必须逐件扫码或录码。
3. 调入仓确认收纸时，必须逐件扫码或录码，并为每件录入调入后的库位。
4. 提纸单不允许部分发纸、部分收纸。
5. 调出扫码件数必须与提纸单件数完全一致，才允许提交发纸。
6. 调入扫码件数必须与调出已发件数完全一致，才允许提交收纸。
7. 调入收货时若扫码到非本单已发出的唯一编号，系统应拦截。
8. 调拨完成后，每件纸张的当前仓库更新为调入仓库，当前库位更新为调入库位。

#### 5.11 调出仓确认发纸规则

1. 仅允许扫描当前处于在库状态、且所在仓库为调出仓库的纸张。
2. 若扫码纸张不在当前调出仓，不允许发纸。
3. 发纸完成后，件级状态更新为“已发出待收货”。
4. 发纸后在调入仓确认收货前，该件不应再被其他单据占用出库。

#### 5.12 调入仓确认收纸规则

1. 仅允许扫描当前提纸单已发出的唯一编号。
2. 收纸时必须填写调入仓下的库位。
3. 收纸完成后，件级状态更新为“在库”。
4. 系统记录本次调拨单号，形成轨迹记录。

#### 5.13 验收标准

1. 调出仓确认发纸时，系统只允许扫描本仓可发纸张。
2. 非本单或非本仓纸张扫描时，系统拦截并提示。
3. 提纸单不允许部分发纸，不允许部分收纸。
4. 调入收货后，件级库存中的仓库、库位均完成更新。
5. 调拨完成后，系统自动生成调拨单，且与件级流转结果一致。

#### 5.14 异常处理

| 异常场景 | 处理方式 | 提示文案 |
| ---- | ---- | ---- |
| 扫到非本仓纸张 | 拦截提交 | 该纸张不在当前调出仓库，无法发纸 |
| 扫到非本单纸张 | 拦截提交 | 该纸张不属于当前提纸单，无法操作 |
| 已发件数不足 | 拦截发纸 | 当前发纸件数不足，不能确认发纸 |
| 已收件数不足 | 拦截收货 | 当前收纸件数不足，不能确认收纸 |
| 收货未录库位 | 拦截收货 | 存在未填写库位的纸张件，请完善后再收货 |
| 扫到已被占用纸张 | 拦截提交 | 该纸张当前状态不可用于提纸，请检查后重试 |

### F003 退纸单件级退纸管理

**功能说明**

在现有退纸单流程中增加件级扫码退纸、扫码收纸、库位落位能力，实现印厂退纸回公司过程的逐件核对与追踪。

**适用流程**

公司纸张管理员提交申请 → 退纸的印厂人员确认退纸 → 公司仓管确认收纸 → 系统自动生成调拨单

#### 5.15 页面/节点改造范围

| 节点 | 端 | 改造内容 |
| ---- | ---- | ---- |
| 公司纸张管理员提交申请 | PC | 可查看拟退回件明细或按原方式申请 |
| 印厂人员确认退纸 | 手机优先，PC兼容 | 逐件扫码退纸出库 |
| 公司仓管确认收纸 | 手机优先，PC兼容 | 逐件扫码收纸并录入库位 |
| 自动生成调拨单 | 系统 | 生成调拨单并更新件级库存 |

#### 5.16 业务规则

1. 退纸单需按件维度确认实际退回的纸张。
2. 退纸印厂确认退纸时，必须逐件扫码或录码。
3. 公司仓管确认收纸时，必须逐件扫码或录码，并填写公司收货仓库下库位。
4. 退纸单不允许部分退纸、部分收纸。
5. 退纸出库件数必须与退纸单件数完全一致，才允许提交退纸。
6. 收纸件数必须与已退纸件数完全一致，才允许提交收纸。
7. 收纸完成后，每件纸张的当前仓库更新为公司收货仓库，当前库位更新为收货库位。

#### 5.17 验收标准

1. 印厂确认退纸时，只允许扫描当前可退回的纸张件。
2. 公司确认收纸时，只允许扫描本退纸单已退回的纸张件。
3. 未完成全部退纸或全部收纸时，系统不允许提交。
4. 收纸完成后，件级库存更新至公司仓及对应库位。
5. 系统自动生成调拨单，且件级流转数据可追溯。

#### 5.18 异常处理

| 异常场景 | 处理方式 | 提示文案 |
| ---- | ---- | ---- |
| 扫到非本单退纸 | 拦截提交 | 该纸张不属于当前退纸单，无法操作 |
| 退纸件数不足 | 拦截退纸 | 当前退纸件数不足，不能确认退纸 |
| 收纸件数不足 | 拦截收货 | 当前收纸件数不足，不能确认收纸 |
| 收纸未填库位 | 拦截收货 | 存在未填写库位的纸张件，请完善后再收货 |
| 纸张状态异常 | 拦截提交 | 该纸张当前状态不允许退纸，请检查后重试 |

### F004 一物一码与标签打印

**功能说明**

提供纸张件级唯一编号生成、二维码生成、标签打印、手工录码补充等基础能力，支撑三类单据的件级管理。

#### 5.19 编码生成规则

1. 系统按自然年维护年度流水。
2. 每年从 `000001` 开始递增。
3. 编码格式固定为 `YYYYNNNNNN`。
4. 示例：`2026000001`、`2026000002`。
5. 同一年度内编码不可重复。
6. 编码一旦生成并用于件明细，不允许随意修改。

#### 5.20 二维码标签打印规则

1. 标签内容仅包含二维码图形和唯一编号文本。
2. 支持按单据批量打印。
3. 支持按件明细单独补打。
4. 标签打印能力覆盖随货同行单新入库场景，也支持历史库存补码场景。
5. 二维码内容为唯一编号本身。

#### 5.21 手工录码规则

1. 所有要求扫码的节点，均支持手工输入唯一编号作为补充方式。
2. 手工输入后按与扫码一致的规则进行校验。
3. 输入不存在、重复、状态不匹配或不属于当前业务范围时，系统拦截并提示。

#### 5.22 验收标准

1. 系统可按年度正确生成连续唯一编号。
2. 每年切换后，流水从 `000001` 重新开始。
3. 可按单据批量打印二维码标签。
4. 可对单件纸张进行二维码补打。
5. 在所有扫码节点均可使用手工录码完成识别和校验。

### F005 库位管理

**功能说明**

在仓库维度下新增库位主数据管理，并在纸张件级库存中落位，支持实物查找和后续调拨。

#### 5.23 业务规则

1. 每个仓库下可配置多个库位。
2. 本期仅支持“仓库 + 库位”两级结构。
3. 纸张完成入库、收货时，必须落到具体库位。
4. 库位作为件级库存字段保存，不作为单据上层汇总字段替代项。
5. 库位权限不在业务内写死，沿用云平台统一角色菜单权限配置。

#### 5.24 验收标准

1. 系统支持维护仓库下库位主数据。
2. 随货同行单入库、提纸单收货、退纸单收货时均可选择库位。
3. 件级库存中可查询每件纸当前所在库位。

### F006 一物一码查询

**功能说明**

提供按唯一编号查询纸张件级信息的能力，支持管理人员、仓管快速定位与追溯。

#### 5.25 查询内容

| 查询项 | 说明 |
| ---- | ---- |
| 唯一编号 | 纸张一物一码编号 |
| 纸张基础信息 | 所属纸张品类/规格等上层信息 |
| 当前状态 | 当前是否在库、已发出待收货等 |
| 当前仓库 | 当前所在仓库 |
| 当前库位 | 当前所在库位 |
| 来源单据 | 最初来源单据 |
| 最近一次业务单据 | 最近一次涉及该件的单据 |
| 流转轨迹 | 入库、调拨、退纸等历史记录 |
| 出厂编码 | 供应商编码，如有 |
| 单件吨数 | 当前件级吨数 |
| 照片 | 如有上传则可查看 |

#### 5.26 业务规则

1. 支持按唯一编号精确查询。
2. 查询结果需展示当前库存定位信息。
3. 查询结果需展示完整流转轨迹，至少包括单据类型、单据编号、操作节点、操作时间。
4. 若唯一编号不存在，返回空并提示。

#### 5.27 验收标准

1. 输入唯一编号后可查看该件纸张当前仓库、库位、状态。
2. 可查看该件纸张的历史单据轨迹。
3. 不存在的唯一编号能明确提示，不返回错误数据。

### F007 历史库存初始化

**功能说明**

支持将线下补码完成的历史库存件级数据，通过初始化方式导入系统，纳入统一库存管理。

#### 5.28 初始化场景

- 历史库存已由纸张管理人员线下打印二维码并贴码。
- 已通过 Excel 维护唯一编号及基础件级信息。
- 需将历史库存一次性或分批导入系统。

#### 5.29 初始化字段建议

| 字段 | 必填 | 说明 |
| ---- | ---- | ---- |
| 唯一编号 | 是 | 历史库存一物一码 |
| 纸张ID/纸张名称 | 是 | 对应系统纸张基础资料 |
| 仓库 | 是 | 当前所在仓库 |
| 库位 | 是 | 当前所在库位 |
| 吨数 | 是 | 单件吨数 |
| 出厂编码 | 否 | 供应商编码 |
| 照片 | 否 | 图片地址或关联信息，具体导入方式待技术确定 |
| 初始化日期 | 否 | 记录导入时间 |
| 备注 | 否 | 特殊说明 |

#### 5.30 初始化校验规则

1. 唯一编号不能为空、不能重复。
2. 纸张基础资料必须能匹配系统内已有纸张。
3. 仓库必须为系统内有效仓库。
4. 库位必须为对应仓库下有效库位。
5. 吨数必须大于 0。
6. 出厂编码不做唯一性校验。
7. 校验失败的数据不得入库，应返回失败原因。

#### 5.31 初始化结果处理

1. 初始化成功后，件级库存状态统一记为“在库”。
2. 初始化成功后，可直接纳入后续提纸、退纸、查询等业务使用。
3. 系统需输出初始化结果汇总，包括成功数量、失败数量、失败原因。

#### 5.32 验收标准

1. 系统支持按模板导入历史库存数据。
2. 校验失败数据可明确反馈原因。
3. 导入成功后，历史库存可在一物一码查询中查到。
4. 导入成功后，历史库存可参与后续扫码流转。

## 11. 结论

本次迭代的核心，是在现有随货同行单、提纸单、退纸单流程不变的前提下，将纸张管理从“纸张规格级”提升到“单件级”，形成一物一码、扫码出入库、仓库库位管理、历史库存纳管和轨迹追溯的完整闭环。完成后，系统将具备以下能力：

1. 每件纸张有唯一身份，可查可追。
2. 所有关键实物流转节点均可扫码核对。
