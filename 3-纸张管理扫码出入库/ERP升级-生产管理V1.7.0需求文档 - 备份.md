## 2. 背景与目标

### 2.1 项目背景

当前 ERP 已支持随货同行单、提纸单、退纸单三类纸张业务流程，但仍主要管理到纸张品类/规格层级，缺少纸张件级管理、一物一码、扫码核对、库位记录和统一追踪能力。历史库存仍主要依赖线下贴码和 Excel 管理。

本次迭代在现有业务流程基础上，新增纸张件级管理、扫码出入库、库位记录、二维码生成打印和历史库存初始化能力。

本期主要用户包括公司仓管、调拨相关仓管、印厂退纸人员、公司纸张管理员及历史库存初始化人员。

### 2.2 本期目标

- 实现纸张件级管理，一件纸对应一个系统唯一编号。
- 在入库、发纸、退纸、收纸节点实现扫码核对和库位记录。
- 提供一物一码查询与轨迹追溯能力。
- 将历史库存初始化进系统，纳入后续统一管理。

### 2.3 本期范围

- 新增纸张件级档案能力，支持单据明细行与纸张件建立一对多关系。
- 随货同行单接入件级建档、扫码入库和库位落位能力。
- 提纸单接入件级发纸、件级收纸和库位落位能力。
- 退纸单接入件级退纸、件级收纸和库位落位能力。
- 提供纸张一物一码编码、二维码生成和打印能力。
- 复用现有仓库库位管理模块，在纸张件台账（即一物一码当前仓库、库位和来源等信息）中记录仓库与库位。
- 随货同行单、提纸单、退纸单形成正式业务结果后，按物料、仓库、库位聚合对接 ERP 仓储模块。
- 提供一物一码查询能力，查询对象为纸张件。
- 支持历史库存线下贴码和后端初始化导入，本期不建设历史库存初始化前端页面。

### 2.4 产品原则

1. 不新增审批主线，按现有业务节点挂接实现。
2. 系统中所有纸张件级流转必须以唯一编号为核心，不允许同一编号对应多件纸。
3. 入库和收货结果必须记录明确仓库和库位。
4. 所有出入库核对环节以逐件扫码为主，手工输入二维码编号为扫码失败时的补充方式。
5. 提纸单、退纸单整单处理，不支持部分发货、部分收货、部分退货。
6. 纸张件侧以结果记录为主，用于查询与追溯。

### 2.5 需求优先级策略

- P0：最高优先级，覆盖提纸单、退纸单相关能力。
- P1：次优先级，覆盖随货同行单、查询追溯、通用基础能力、历史库存初始化等其他能力。
- 执行策略：本期仅按 P0/P1 两档区分业务优先级；技术实现中仍需优先打通件级编号、纸张件台账、结果记录字段、库位落账和二维码生成打印等共用基础能力。

### 3.2 核心方案说明

#### 3.2.1 最小管理单元

- 卷筒纸：按“一卷”作为一件管理。
- 平板纸：按“一件”作为一件管理。
- 系统内统一抽象为“纸张件”。

#### 3.2.2 一物一码规则

- 编码格式：`年份 + 6位流水号`。
- 流水号按年份递增。
- 每个自然年从 `000001` 重新开始。
- 唯一编号在系统内全局唯一，同一年内不得重复。

#### 3.2.3 二维码标签规则

- 标签内容包含：二维码、唯一编号、物料名称/规格、吨数、出厂编码。
- 二维码识别结果与手工输入的唯一编号口径一致。
- 标签贴附时机：在入库前完成生成和打印，并贴于对应纸张实物。
- 同一件纸在不同入口打印时，均沿用原唯一编号。

#### 3.2.4 库位规则

- 库位模型为“仓库 + 库位”两级。
- 一件纸张在入库完成后必须落到明确库位。
- 调入收货、退纸收货等再次入库场景，均需重新确认库位。
- 收纸类节点提供单据级“默认库位”能力；默认库位用于承接当前选中的库位或扫库位结果，并作为后续新扫码件的默认带入值。
- 件级库位默认继承扫码确认当时的单据级默认库位，允许按件单独调整。
- 作业过程中如再次修改单据级默认库位，仅影响后续新扫码处理的件，不影响此前已扫码或已确认件的库位。
- 默认库位和件级库位都必须来自当前业务仓库下的有效库位。
- 提纸单“确认发纸”和退纸单“确认退纸”节点增加“当前仓库库位件数”辅助查询入口，用于在作业前查看当前仓库下各物料的库位件数分布，辅助拣货和出库。
- “当前仓库库位件数”按当前单据涉及物料汇总查询，并以弹窗形式集中展示；同一弹窗内支持按物料分组展示多个物料的当前库位件数。
- 该能力仅用于查询展示，不替代件明细中的“调入库位”口径，不新增锁库、占库、提交流程或扫码校验；未维护当前库位的库存统一展示为“未维护当前库位”。

#### 3.2.5 扫码规则

- 随货同行单在“编辑”环节完成件级建档和二维码打印；入库形式为“直送印厂”时，在“印厂确认到货”环节执行扫码或手工录码，入库形式为“公司”时，在“仓库确认入库”环节执行扫码或手工录码。扫码或录码的编号必须属于当前单据在“编辑”环节已建档的件明细编号。
- 提纸单在“确认发纸”节点执行扫码或手工录码，扫一个新增一件明细，扫码或录码形成本次实际发纸件明细。
- 提纸单在“确认收纸”节点执行扫码或手工录码，扫一个确认一个；扫码或录码的编号必须属于本单已发纸件明细。
- 退纸单在“确认退纸”节点执行扫码或手工录码，扫一个新增一件明细，扫码或录码形成本次实际退纸件明细。
- 退纸单在“确认收纸”节点执行扫码或手工录码，扫一个确认一个；扫码或录码的编号必须属于本单已退纸件明细。
- 扫码失败时支持手工输入唯一编号。
- 手工输入与扫码共用同一套校验逻辑，支持输入后按回车或软键盘“确认/完成”触发校验。
- 提纸单、退纸单手机端作业页扫码成功后不自动弹出件明细弹窗，页面停留在扫码区，便于连续扫码。
- 收纸类节点如已选择或扫入默认库位，则后续新扫码确认的件默认继承当前默认库位；中途切换默认库位仅影响后续新扫码件。
- 扫码或手工输入的唯一编号均需校验编号有效性。
- 不允许使用未建码纸张完成正式件级流转。

#### 3.2.6 异常处理规则

| 场景             | 处理规则                           | 预期结果                         |
| ---------------- | ---------------------------------- | -------------------------------- |
| 重复扫码同一码   | 同一作业单内再次扫描同一码必须拦截 | 提示重复扫码，不重复计数         |
| 扫到非本单据件   | 必须按当前单据件明细或业务范围校验 | 提示“非本单据件”，不得加入明细 |
| 手工录码不存在   | 校验唯一编号存在性                 | 提示编号不存在                   |
| 库位未填写即提交 | 入库/收纸类节点强校验              | 拦截提交                         |
| 作业提交失败     | 失败前写入不得视为最终结果         | 不得写入错误的正式结果记录       |

## 4. 功能详情

本章按 PC端、手机端、配套能力与实施展开；前 2 类章节顺序与 3.3 功能清单保持一致。配套能力与实施用于承接通用能力和上线配套动作。

手机端定位为现场作业端，PC端定位为管理操作端；具体节点能力、页面行为和字段要求以本章为准。

### 4.1 PC端功能详情

#### 4.1.1 件明细（随货同行单-PC）

**对应功能清单**：F001-0

**功能说明**

PC端在随货同行单各业务节点提供统一的件明细展示区，用于按物料查看当前单据下已建档的纸张件信息，为编辑维护、审核核对和后续确认入库提供统一的件级视图。

**业务规则**

1. 随货同行单件明细按物料分组展示，不允许将不同物料的件明细混在同一列表中查看或维护。
2. “详情”“编辑”“审核”“印厂确认到货”“仓库确认入库”页面均展示件明细区。
3. “纸张信息”区域先展示“物料汇总表”，再展示“件明细”；件明细位于物料汇总表下方。
4. PC端不同物料的件明细采用同页按物料分组折叠展示。
5. 单物料场景默认展开该物料件明细；多物料场景默认展开首个物料分组，其余分组收起，用户点击分组标题切换查看。
6. “详情”“审核”“印厂确认到货”“仓库确认入库”页面的件明细均为只读展示。

**界面要素**

| 元素             | 类型         | 说明                                                                                                                                                                 |
| ---------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 件明细小标题     | Text         | 位于“纸张信息”区域内，紧跟“物料汇总表”之后展示，用于标识下方为按物料分组的件明细区。                                                                                 |
| 物料件明细折叠组 | CollapseItem | 位于“件明细”小标题下方；每条物料明细对应一个折叠组，标题展示物料名称和件数，点击后展开或收起当前物料件明细。                                                         |
| 件明细表格       | Table        | 位于对应物料件明细折叠组内；用于展示该物料下的全部件明细。统一展示唯一编号、入库数量、吨数、出厂编码、图片、当前库位等字段。                                         |

**异常处理**

| 异常场景                 | 处理方式  | 提示文案                     |
| ------------------------ | --------- | ---------------------------- |
| 物料明细下无件明细       | 页面提示  | 当前物料下暂无件明细         |

#### 4.1.2 编辑单据（随货同行单-PC）

**对应功能清单**：F001-1

**功能说明**

PC端“编辑单据”用于为随货同行单预先建立件级档案，为审核通过后的唯一编号生成、二维码打印、移动端入库扫码和追溯查询提供基础数据。

**业务规则**

1. 用户从“已整理单据(纸张)”列表点击“编辑”进入当前随货同行单详情编辑页。
2. 编辑页“纸张信息”区域先展示“物料汇总表”，再展示“件明细”；件明细按物料分组折叠展示。单物料默认展开，多物料默认展开首个物料分组，且同一时刻仅展开一个分组。
3. “新增件明细”仅支持在当前展开物料分组下逐行新增，不支持按件数批量生成空白行。
4. 件明细支持删除；删除后当前物料分组内件序号自动重排。
5. 件明细至少包含唯一编号、入库数量、吨数、出厂编码、图片、库位等字段；其中吨数、入库数量、出厂编码、图片、库位由用户维护。唯一编号在编辑阶段不生成，由审核通过后系统批量生成。
6. 用户录入吨数后，系统读取当前物料档案中的转换率，按“入库数量=吨数×转换率”自动回算入库数量；如用户手动修改入库数量，系统不反算吨数。
7. 单据头“库位”可编辑；用户修改单据头库位后，系统自动同步全部件明细库位。件明细库位也支持逐行编辑；如后续再次变更单据头库位，则全部件明细按最新单据头库位重新同步。
8. 编辑页中吨数、入库数量、出厂编码、图片、库位可编辑；唯一编号在当前页面不编辑，审核通过前为空，审核通过后回填显示。图片字段支持多图上传，已上传图片以缩略图列表展示，并支持继续上传和删除。
9. 页面顶部操作区提供“暂存”和“提交审核”两个按钮；“暂存”仅保存当前编辑结果，不推进流程，“提交审核”在校验通过后保存并提交审核。当前单据下各物料均需补齐件明细。
10. 提交审核后，单据进入待审核状态；审核通过前允许按现有流程撤回审核，且系统不生成唯一编号。审核通过时，系统按单据一次性为全部件明细生成唯一编号；仅审核通过且已生成唯一编号的单据，才可在列表页打印二维码、执行移动端确认入库和进行后续一物一码查询。

| 元素             | 类型         | 说明                                                                                                                                                                                                                                         |
| ---------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 物料件明细折叠组 | CollapseItem | 位于“件明细”区域内；每条物料明细对应一个折叠组，默认首组展开，用户点击分组标题切换查看。                                                                                                                                                         |
| 件明细表格       | Table        | 位于对应物料件明细折叠组内；展示件序号、唯一编号、入库数量、吨数、出厂编码、图片、库位、操作等字段。吨数、入库数量、出厂编码、图片、库位在当前页面可编辑；唯一编号在当前页面不编辑，审核通过前为空，审核通过后回填显示。                     |
| 新增件明细按钮   | Button       | 位于物料件明细折叠组标题右侧；每点击一次仅在当前物料分组下新增一条空白件明细，不支持批量新增、不支持跨物料新增。                                                                                                                                   |

#### 4.1.3 打印二维码（随货同行单-PC）

**对应功能清单**：F001-2

**功能说明**

PC端在“已整理单据(纸张)”列表提供二维码打印能力。用户对审核通过且已生成唯一编号的随货同行单点击“打印二维码”后，进入当前单据下全部件明细二维码的打印页。

**业务规则**

#### 4.1.7 确认发纸（提纸单-PC）

**对应功能清单**：F002-1

**功能说明**

当手机端扫码不可用时，PC端提供提纸单确认发纸的兜底处理能力。用户在当前页面按物料分组新增件明细、录入唯一编号，系统回填只读件级字段，并完成发纸确认。

**业务规则**

1. 页面结构为“单据信息”在上、“提纸明细”在下；“单据信息”区域展示单据编号、状态、提交时间、调出地点、调入地点、调入库位、物料分类等字段，其中单据级“调入库位”可编辑。
2. “提纸明细”区域按物料分组展示“物料汇总表 + 件明细手风琴分组”。每个物料分组标题展示“应发件数 / 已录件数”。
3. 页面在“提纸明细”标题右侧提供“当前仓库库位件数”入口；点击后打开弹窗，按物料分组展示当前调出仓下各物料的库位件数分布。弹窗标题统一为“当前仓库库位件数”，说明文案需明确“展示本单各物料在当前仓库下的库位件数，不含调入库位信息”。
4. “当前仓库库位件数”弹窗统一展示当前仓库名称、物料名称、库位数/件数汇总、当前库位、件数；长物料名称允许换行展示；未维护当前库位的库存统一展示为“未维护当前库位”。
5. 该弹窗仅用于查询展示，不影响“新增件明细”、唯一编号校验、提交校验和调入库位维护逻辑。
6. 用户点击“新增件明细”后，仅在当前物料分组下新增一条空白件明细；当前物料分组已达到应发件数时，不允许继续新增。
7. 件明细行统一展示件序号、唯一编号、出厂编码、调拨数量、吨数、图片、操作；不展示调入库位。其中件序号、出厂编码、调拨数量、吨数、图片只读；唯一编号可维护。
8. 用户录入唯一编号后，系统立即校验编号是否存在、当前所在仓库是否为本单调出仓、纸张件物料编码是否存在于当前单据原料明细中、是否已被本单重复录入。校验通过后自动回填该件的出厂编码、调拨数量、吨数，并同步展示纸张件档案中的件级图片；校验失败时清空当前行派生字段并提示原因。
9. 图片列仅展示纸张件档案中的件级图片，不支持上传、继续上传、补图或删除。
10. 用户可删除当前物料分组下的件明细；删除后当前物料分组内件序号自动重排。
11. 仅当全部物料分组“已录件数=应发件数”且所有件明细校验通过后，才允许点击“提交”；“暂存”仅保存当前页面已录入内容，不推进流程。
12. 提交成功后写入发纸结果、结果单据编号和操作日志；失败、取消或仅暂存不得形成正式发纸结果。

#### 4.1.12 一物一码查询（PC）

**对应功能清单**：F006-1

**功能说明**

PC端提供纸张件维度的查询追溯页面，采用单页面查询结构，在同一查询区平铺展示唯一编号、原料编码、原料名称三个查询条件；原料名称为选择型查询条件，支持输入关键字模糊匹配候选项后选择；用户输入或选择查询条件后，可查看件明细列表、选中件详情和单据结果记录。

**业务规则**

1. 页面按“查询区 -> 件明细列表 -> 选中件详情 -> 单据结果记录”组织，查询对象为纸张件。
2. 查询条件支持唯一编号、原料编码、原料名称单独或组合查询；原料编码对应唯一纸张资料，原料名称通过关键字模糊匹配候选项并选择后作为精确查询条件；点击“查询”后按当前已填写条件返回结果，点击“重置”后清空全部查询条件并恢复空态。

5. 单据结果记录表仅展示正式业务结果，不展示草稿、撤回、驳回等未形成正式结果的过程数据。
6. 无查询条件、点击重置或未查询到数据时，页面展示统一空态提示，不展示件明细列表、选中件详情和单据结果记录表。
7. 字段取值规则如下：

   - 单据结果记录表：序号按当前记录展示顺序从1开始生成；结果时间、单据类型、单据编号、结果均取单据级结果记录中的正式结果字段。

#### 4.2.2 确认入库（随货同行单-手机）

**对应功能清单**：F001-6

**功能说明**

手机端用于在随货同行单的“印厂确认到货”或“仓库确认入库”节点，对实际入库的纸张件逐件扫码或录码完成正式入账。

**业务规则**

1. 入库形式为“直送印厂”时，在“印厂确认到货”节点执行确认入库；入库形式为“公司”时，在“仓库确认入库”节点执行确认入库。
2. 页面结构按“状态卡 -> 随货同行单图片 -> 确认入库作业区 -> 原料明细 -> 件明细 -> 单据信息 -> 底部按钮”组织，其中确认入库作业区为整单级作业入口。
3. 当前节点仅允许对本单已生成、且处于待确认状态的件明细执行确认入库。用户可通过扫码，或人工录码后点击软键盘“确认/完成”；两种方式均执行同一套校验和确认逻辑，校验通过后立即确认对应件。
4. 仅允许确认属于当前单据的件明细，同一件不得重复确认。扫码或录码成功后，输入框自动清空并保持可继续输入状态；页面以轻提示方式反馈“识别成功，已确认XXX”，同步更新该件确认状态、已确认/待确认数量和提交按钮状态，并高亮命中的件明细；已确认且尚未提交的件，支持在件明细中执行“取消确认”，取消后该件确认状态回退为“待确认”，并同步回刷进度统计和提交按钮状态。
5. 用户可在确认入库作业区按整单选择当前入库仓下默认库位，也可通过“扫库位”快速带入；默认库位在选择或扫码后立即同步至当前全部件明细。
6. 件明细保留件级库位调整入口，用户可对个别件单独调整库位；件级库位支持选择和扫库位，且默认继承默认库位；如后续再次调整默认库位，则全部件明细按最新默认库位重新同步，并覆盖此前已手工调整的件级库位。
7. 默认库位及调整后的件级库位必须均来自当前入库仓下的有效库位；仅当应入件全部确认完成且每一件明细均已维护有效库位时，才允许提交。
8. 提交成功后记录入库结果和操作日志，并写入当前仓库、当前库位和来源单据。

| 默认库位选择器   | Select | 位于确认入库作业区顶部；仅加载当前入库仓下有效库位，选择后立即同步至全部件明细。                                                                                                              |
| 默认库位扫码按钮 | Button | 位于默认库位选择器右侧；点击后调起摄像头扫描库位码，识别成功后自动选中对应默认库位。                                                                                                                     |

#### 4.2.3 件明细（提纸单-手机）

6. “确认收纸”页的件明细卡片额外展示件级确认状态，状态文案统一为“待确认/已确认”。
7. 件明细弹窗在卡片下方提供“库位件数统计”区域，用于统计当前单据件明细分布在哪些库位；该区域默认可折叠，展开后逐行展示“库位名称 + 件数”，不使用进度条或比例图。

6. 用户扫码或录码后，系统校验唯一编号是否存在、是否属于当前退纸单允许退纸范围、是否被本单重复录入；校验通过后，在对应物料下新增件明细，并自动回填出厂编码、调拨数量、吨数和件级图片。仅允许处理当前可退回且件级数据完整的纸张件。
7. 页内默认提示文案为“等待扫码或输入编号后回车确认”；扫码或录码成功后，页内提示更新为“识别成功，已新增退纸件：{唯一编号}”，录入最后一件时更新为“识别成功，已录入最后一件退纸：{唯一编号}”。成功后清空输入框、刷新进度统计，并保持页面停留在扫码录码区，便于连续扫码；扫码成功后不自动弹出件明细弹窗。
8. 新增件明细通过原料明细右侧“件明细（N）”入口查看；弹窗中的件明细卡片遵循 `4.2.6 件明细（退纸单-手机）` 定义，不展示件序号，展示唯一编号、出厂编码、调拨数量、吨数、调入库位、图片。
9. 对当前节点已新增但尚未提交的件明细，支持在件明细弹窗卡片底部执行“删除件明细”；删除后该件立即从当前物料件明细中移除，同步回刷应退/已录/待录统计和件数标签。删除仅影响本次确认退纸节点下尚未提交的临时件明细，不影响其他单据或已形成正式结果的件记录。
10. 被删除的件明细，如唯一编号仍满足本单校验范围，允许再次扫码或录码重新加入；重新加入后继续参与提交校验。

#### 4.2.8 确认收纸（退纸单-手机）

**功能说明**

手机端在退纸单“确认收纸”节点通过页面顶部整单级扫码录码区逐件扫码或录码确认，支持整单默认调入库位和件级调入库位调整，扫一个确认一个，并在确认区维护纸张到库信息。

**业务规则**

1. 页面采用“状态卡 -> 扫码录码区 -> 原料明细 -> 单据信息 -> 确认信息 -> 底部按钮”的结构；扫码录码区位于原料明细上方，采用紧凑布局，不单独展示“确认收纸”标题文案，包含默认调入库位、扫库位图标、唯一编号输入、扫码主按钮、进度统计和页内提示。
2. 仅允许按本单已生成的件明细逐件扫码或录码完成确认；系统校验命中件是否属于当前退纸单已退纸件，校验通过后将该件确认状态更新为“已确认”，再次扫描已确认件时提示“当前纸张已确认，请勿重复操作”；已确认且尚未提交的件，支持在件明细卡片底部执行“取消收纸”，取消后该件确认状态回退为“待确认”，并同步回刷进度统计和提交按钮状态；该确认状态仅用于当前页面扫码确认进度展示和提交校验，不作为件明细正式状态字段单独维护。
3. 页面支持按单据级选择默认调入库位，也支持通过扫库位图标识别并选中默认库位；默认库位和件级库位都必须来自当前调入仓下有效库位。
4. 当前已选择或扫入的默认库位，仅作用于后续新扫码确认的件；作业过程中如切换默认库位，不影响此前已扫码或已确认件的库位。每个件明细仍支持单独调整调入库位。
5. 扫码区需展示明确提示文案，说明“上方库位仅作为后续扫码件的默认库位；切换库位后，不影响已扫码件的库位，之后新扫码的件按新库位带入”。
6. 页内默认提示文案为“等待扫码或输入编号后回车确认”；扫码或录码成功后，页内提示更新为“识别成功，当前纸张已确认：{唯一编号}”，确认最后一件时更新为“识别成功，已完成最后一件确认：{唯一编号}”。成功后清空输入框、刷新进度统计，并保持页面停留在扫码录码区，便于连续扫码；扫码成功后不自动弹出件明细弹窗。
7. 件明细通过原料明细右侧“件明细（N）”入口查看；弹窗标题后展示当前序号/总数，卡片支持左右滑动和左右按钮切换。卡片统一不展示件序号，展示确认状态、唯一编号、出厂编码、调拨数量、吨数、调入库位、图片，不展示调入仓库；已确认件在卡片底部展示“取消收纸”按钮。
8. 件明细弹窗在卡片下方提供“库位件数统计”区域，默认可折叠；展开后展示当前单据各库位对应件数，便于快速查看库位分布。
9. 页面底部确认信息区维护纸张到库时间和纸张到库图片；仅当全部件确认状态均为“已确认”、每一件明细均已维护有效调入库位，且纸张到库时间和纸张到库图片均已维护时允许提交。退纸单底部左侧操作按钮沿用既有“返单”文案。

#### 4.2.9 一物一码查询（手机）

**对应功能清单**：F006-2

**功能说明**

手机端提供纸张件维度的查询追溯入口，支持扫码或手工输入唯一编号，也支持按原料编码、原料名称筛选候选件；原料名称为选择型查询条件，支持输入关键字模糊匹配候选项后选择；查询结果以“物料信息 + 件明细列表”方式展示，便于现场快速定位目标纸张件。

**业务规则**

1. 页面采用“查询条件 -> 物料信息 -> 件明细列表”的单页结构；查询条件支持唯一编号、原料编码、原料名称单独或组合查询，其中唯一编号支持扫码带入，原料名称支持关键字模糊匹配候选项并选择后查询。
2. 用户点击“查询”后按当前已填写条件返回结果；点击“重置”后清空全部查询条件并恢复空态，不保留上一轮查询结果。
3. 当查询命中结果时，页面展示物料信息区和件明细列表；件明细候选卡片支持点击进入对应纸张件的手机端详情页。
4. 当无查询条件、点击重置或查询无结果时，页面仅展示空态提示，不展示物料信息区和件明细列表。
5. 字段取值规则如下：
   - 查询条件区：唯一编号支持扫码结果与手工录码混用，扫码识别结果与手工输入口径一致；原料编码对应唯一纸张资料；原料名称通过关键字模糊匹配候选项并选择后作为精确查询条件。
   - 物料信息区：原料编码、原料名称取当前查询条件命中的唯一纸张资料；查询结果中的件明细必须归属于该纸张资料。
   - 件明细列表：件数按当前查询结果命中的纸张件数量统计；仓库数按当前结果中去重后的当前仓库数量统计；卡片中的唯一编号取纸张件档案唯一编号，当前仓库、当前库位取纸张件台账中的当前生效仓库和当前生效库位。

#### 4.2.10 一物一码件详情（手机）

**对应功能清单**：F006-3

**功能说明**

手机端提供纸张件详情页，用于查看单件基础信息、当前所在仓库/库位，以及该纸张件关联的正式单据结果记录。

**业务规则**

1. 页面采用“顶部摘要 -> 基础信息 -> 单据结果记录”的结构，用于查看单件当前状态和历史正式结果；本页面只读，不提供编辑入口。
2. 页面支持通过顶部返回或关闭操作回到上一页；当存在上一页历史记录时返回查询结果页，否则回到手机端一物一码查询首页。
3. 单据结果记录只展示正式业务结果，不展示草稿、驳回、撤回等未形成正式结果的过程数据。
4. 字段取值规则如下：
   - 顶部摘要区：唯一编号取当前纸张件档案唯一编号；当前仓库、当前库位取纸张件台账中的当前生效仓库和当前生效库位。
   - 基础信息区：唯一编号、吨数、出厂编码、件级图片取纸张件档案；原料编码、原料名称取关联纸张资料；存储数量按当前纸张件对应业务场景取值，随货同行单入库场景取入库数量，提纸单、退纸单等流转场景取调拨数量；当前仓库、当前库位取纸张件台账中的当前生效仓库和当前生效库位。
   - 单据结果记录区：记录标题按当前结果记录显示顺序生成“记录1、记录2……”；结果时间、单据类型、单据编号、结果均取单据级结果记录中的正式结果字段。

**界面要素**

| 元素           | 类型   | 说明                                                                                          |
| -------------- | ------ | --------------------------------------------------------------------------------------------- |

### 4.3 配套能力与实施

#### 4.3.1 库位接入与校验

**对应功能清单**：F005-1、F005-2

**功能说明**

纸张业务直接复用现有仓库库位主数据和管理模块；系统统一校验仓库与库位的匹配关系，保证业务页面只能选择当前仓库下的有效库位。

**业务规则**

1. 单据节点调用现有仓库库位管理模块，纸张业务直接使用现有仓库库位主数据。
2. 前端加载可选库位时按当前业务仓库过滤，使用现有仓库库位主数据渲染可选项。
3. 仅允许选择当前仓库下有效状态的库位；非当前仓库或无效状态的库位不允许被选择。
4. 提交时后端仍需执行仓库与库位的二次校验。
5. 库位主数据获取失败时，在当前业务页提示并阻止继续选择库位。

**界面要素**

| 元素             | 类型   | 说明                                                                                                                        |
| ---------------- | ------ | --------------------------------------------------------------------------------------------------------------------------- |
| 业务页库位选择器 | Select | 出现在各入库页及件明细形成页的默认库位选择器和件级库位选择器中；使用现有仓库库位主数据，根据当前仓库动态加载可选有效库位。 |
| 字段校验提示     | Toast  | 当用户选择无效库位、仓库与库位不匹配或提交时存在非法组合时给出拦截提示。                                                    |
| 库位加载失败提示 | Toast  | 当现有库位主数据加载失败时，在当前业务页提示并阻止继续选择库位。                                                            |

**异常处理**

| 异常场景           | 处理方式  | 提示文案                     |
| ------------------ | --------- | ---------------------------- |
| 选择了无效库位     | Toast提示 | 当前库位无效，请重新选择     |
| 仓库与库位不匹配   | Toast提示 | 当前库位不属于所选仓库       |
| 库位主数据获取失败 | Toast提示 | 库位数据加载失败，请稍后重试 |

#### 4.3.2 ERP仓储模块对接

**对应功能清单**：F005-3

**功能说明**

纸张业务在随货同行单、提纸单、退纸单形成正式业务结果后，需要将本次业务对应的库存变动结果同步至 ERP 仓储模块；对接粒度从原“单据内同一物料一条明细”调整为“按物料、仓库、库位聚合后的多条明细”。

**业务规则**

1. ERP 仓储模块对接以本次单据成功件明细为来源，不得直接按当前库存余额重算后覆盖仓储模块数据。
2. 对接前需将本次成功件明细按“物料编码 + 仓库 + 库位”聚合；同一物料存在多个库位时，必须拆分为多条对接明细，不允许合并为一条。
3. 对接数量口径为本次业务变动量；随货同行单取本次确认入库成功件明细对应的入库数量，提纸单、退纸单按各自正式结果对应的调拨数量口径取值。
4. 单据内同一物料、同一仓库、同一库位下的多件纸张，可在对接 ERP 仓储模块前汇总为一条变动明细；纸张业务侧仍保留件级台账和件级结果记录，不因对接聚合而丢失件级追溯能力。
5. 仅当单据形成正式业务结果后才允许触发 ERP 仓储模块对接；草稿、驳回、撤回、提交失败等未形成正式结果的场景不得生成正式对接数据。
6. 如 ERP 仓储模块对接失败，纸张业务侧需保留失败日志和失败原因，避免误认为已完成仓储同步；是否支持自动重试或人工补同步，后续结合实施方案单独明确。

**界面要素**

| 元素           | 类型 | 说明                                                                 |
| -------------- | ---- | -------------------------------------------------------------------- |
| 对接结果日志   | Log  | 用于记录每次 ERP 仓储模块对接的触发时间、触发单据、成功状态和失败原因。 |
| 对接明细数据集 | Data | 由本次成功件明细按物料、仓库、库位聚合后生成，作为 ERP 仓储模块入参。 |

**异常处理**

| 异常场景             | 处理方式  | 提示文案                         |
| -------------------- | --------- | -------------------------------- |
| 存在空仓库或空库位   | 拦截对接  | 存在未维护完整仓库或库位的件明细 |
| 聚合后明细生成失败   | 记录日志  | 仓储对接明细生成失败             |
| ERP 仓储模块对接失败 | 记录日志  | ERP仓储模块同步失败，请稍后重试  |

#### 4.3.3 初始化导入

**对应功能清单**：F007-3

**功能说明**

本期纳入历史库存初始化导入工作，用于将存量纸张件逐步纳入统一件级管理体系。

**业务规则**

1. 本期明确需要支持历史库存初始化导入工作。
2. 初始化导入的目标是将存量纸张件纳入统一件级管理，并支撑后续业务使用。
3. 具体执行方式、数据模板、字段口径、校验规则、操作流程和结果反馈机制待后续结合实施方案单独确定。

#### 4.3.4 线下贴码

**功能说明**

线下贴码用于在系统正式扫码前完成纸张件二维码实体标签准备，覆盖新入库件和历史库存补码场景。

| 唯一编号文本 | Text  | 与二维码图形同时打印，便于人工核对。 |

## 5. 数据与结果设计

### 5.1 逻辑实体设计

| 实体             | 说明                                                               | 与现有代码关系                 |
| ---------------- | ------------------------------------------------------------------ | ------------------------------ |
| 纸张件档案       | 以唯一编号为主键的件级对象，承载吨数、出厂编码、图片等单件属性     | 现有单据明细行之上新增逻辑对象 |
| 纸张件台账       | 记录纸张件当前仓库、当前库位、来源单据、最近一次结果节点等结果数据 | 为纸张业务侧结果台账           |
| 单据级结果记录   | 记录纸张件关联单据形成的正式结果，如入账、发纸、退纸、收纸等     | 作为一物一码查询数据来源       |
| 单据行与件的关联 | 记录单据明细行与纸张件的一对多关系，支撑汇总与件级下钻             | 保留现有单据汇总口径           |

### 5.2 纸张件核心字段

| 字段             | 说明                                       |
| ---------------- | ------------------------------------------ |
| 唯一编号         | 一物一码主键标识，格式为“4位年份 + 6位数字流水号” |
| 纸张ID           | 对应上层纸张资料                           |
| 纸张名称/规格    | 用于展示和查询                             |
| 单件吨数         | 卷筒纸场景用于业务识别和核对               |
| 入库数量         | 随货同行单入库场景下记录的业务数量字段     |
| 调拨数量         | 提纸单、退纸单等流转场景使用的业务数量字段 |
| 存储数量         | 查询详情页展示字段；按当前纸张件对应业务场景取值，随货同行单入库场景取入库数量，提纸单、退纸单等流转场景取调拨数量 |
| 出厂编码         | 供应商单件编码                             |
| 当前仓库         | 当前所在仓库                               |
| 当前库位         | 当前所在库位                               |
| 来源单据类型     | 首次入账来源，如随货同行单/初始化          |
| 来源单据编号     | 首次入账单号                               |
| 最近结果单据类型 | 最近一次形成结果的业务单据类型             |
| 最近结果单据编号 | 最近一次形成结果的业务单号                 |
| 最近一次结果节点 | 最近一次形成结果的业务节点名称             |
| 最近结果时间     | 最近一次结果写入时间                       |
| 图片             | 件级图片信息                               |

### 5.3 结果记录原则

- 单据审批状态沿用现有系统状态机，单据当前节点表达业务处理阶段。
- 纸张件侧只记录查询与追溯所需的结果信息，不单独扩展复杂过程状态。
- 整单处理场景下，件台账重点记录当前仓库、当前库位、来源单据和最近一次结果节点。

### 5.4 建议结果记录口径

| 结果项   | 说明                                   | 典型场景                           |
| -------- | -------------------------------------- | ---------------------------------- |
| 入账结果 | 记录纸张件已完成首次入账               | 随货同行单入库完成、初始化导入完成 |
| 发纸结果 | 记录纸张件已完成确认发纸               | 提纸单确认发纸                     |
| 退纸结果 | 记录纸张件已完成确认退纸               | 退纸单确认退纸                     |
| 收纸结果 | 记录纸张件已完成确认收纸并更新当前位置 | 提纸单确认收纸、退纸单确认收纸     |

### 5.5 数据落库边界与实现约束

- 本期必须具备以下逻辑对象：纸张件档案、纸张件台账、单据级结果记录、单据行与件关联关系。
- 开发可结合现有架构选择拆表或合表实现，但不得缺失上述业务语义。
- 形成正式业务结果时，必须至少写入最近一次结果节点、最近结果时间和操作日志；需要追溯业务来源的场景，还应写入来源单据或结果单据。
- 首次入账、确认收纸等会改变纸张件当前位置的节点，必须同步更新当前仓库和当前库位。
- 对接 ERP 仓储模块时，必须以本次成功件明细为基础，按物料、仓库、库位聚合生成库存变动明细；同一物料涉及多个库位时，必须拆分多条对接明细，不得按库存余额汇总覆盖。
- 一物一码查询至少依赖纸张件台账与单据级结果记录，不允许只查单据表临时拼装为唯一查询来源。
- 任何失败、取消、返单、驳回场景，如未形成正式业务结果，不得保留错误的正式结果记录。
- 纸张业务侧保留件级台账与件级追溯能力；ERP 仓储模块侧按聚合后的库存变动明细进行对接，不要求按件逐条同步库存。

## 6. 非功能需求

### 6.1 性能要求

| 指标           | 要求       | 说明                             |
| -------------- | ---------- | -------------------------------- |
| 扫码识别反馈   | 1秒内      | 识别后需快速返回结果             |
| 单据件明细加载 | 3秒内      | 常规单据件明细列表加载           |
| 一物一码查询   | 2秒内      | 精确查询当前结果与轨迹           |
| 初始化导入校验 | 可批量执行 | 支持后端执行常规历史库存批量导入 |

### 6.2 安全与审计要求

- 所有件级流转操作需记录操作人、操作时间、操作节点。
- 一物一码查询需受云平台权限控制。
- 初始化导入需保留导入人和导入结果日志。

### 6.3 可用性要求

- 扫码失败时必须可切换到手工录码，不因设备摄像头或识别问题阻塞业务。
- 异常提示文案需明确，可指导用户纠正数据问题。
